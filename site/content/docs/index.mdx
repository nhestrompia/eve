---
title: What is eve?
description: git tracks code, eve tracks product.
---

eve is a product-history layer for Git repositories.

Git is excellent at preserving implementation history: commits, diffs, branches, authorship, and merges. It does not naturally answer the product questions that appear later:

- What changed for users?
- Why was this decision made?
- Which command, screenshot, log, or URL proved the behavior?
- What should a future maintainer or agent know before touching this again?

eve records that missing layer beside Git. Each snapshot describes one completed product change and stores the evidence needed to trust it.

## Why it exists

Modern software work is scattered across pull requests, chat, terminals, screenshots, issue trackers, and agent sessions. The code may be merged, but the context that explains the change often disappears.

eve keeps the important part in the repository:

- a short product summary
- the user-visible change
- decisions and risks
- validation commands and results
- artifacts such as screenshots, logs, URLs, and notes
- the Git branch, commits, and clean or dirty state behind the work

Because snapshots live under `.eve/`, they can be cloned, reviewed, diffed, and carried forward like any other repository file.

## Core idea

One eve snapshot represents one completed unit of product work. It can be a feature, bug fix, experiment, refactor, or release.

```text
Git commit    What changed in the code?
eve snapshot  What changed in the product, why, and how was it verified?
```

eve does not replace Git. It complements Git with product meaning.

## Who it is for

eve is useful for repositories where humans and coding agents both need durable context:

- maintainers reviewing why a feature exists
- contributors working from a fork
- release owners collecting proof of what shipped
- agents that need local memory before editing code
- teams that want product history to travel with the repository

The default workflow is local-first and Git-friendly: implement the change, verify it, commit the implementation, record the eve snapshot, then commit the `.eve/` record.
