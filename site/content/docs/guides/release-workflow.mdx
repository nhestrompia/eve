---
title: Release Workflow
description: How tagged versions become GitHub releases.
---

eve uses Git tags as release triggers.

## Versioning

The CLI version is defined in `types.go` as `CLIVersion`.

Before cutting a release:

1. Update the version if needed.
2. Run the verification suite.
3. Commit the release preparation.
4. Tag the release.
5. Push the tag.

```sh
git tag v0.2.0
git push origin v0.2.0
```

## Release artifacts

The release workflow builds CLI binaries for common operating systems and CPU architectures, then publishes them to a GitHub release.

## Release notes

Use snapshots to draft product-facing release notes:

```sh
eve changelog --since <previous-snapshot-id> --markdown
```

You can also generate notes from a date or an explicit snapshot range:

```sh
eve changelog --since 2026-07-01 --markdown
eve changelog --from <from-snapshot-id> --to <to-snapshot-id> --markdown
```

The changelog uses each snapshot's `userVisibleChange` when present and falls back to the snapshot title.

## Verification

The CI workflow runs:

- `go test ./...`
- `npm --prefix ui ci`
- `npm --prefix ui test`
- `npm --prefix ui run build`
- `npm --prefix site ci`
- `npm --prefix site run build`

Treat a green CI run as the minimum bar. If the release includes a product-facing change, record the targeted verification in the eve snapshot as well.
