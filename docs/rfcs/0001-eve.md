# RFC-0001: EVE Engineering Evolution Protocol

Status: Draft
Version: 0.1

## Abstract

Git records implementation history. EVE records engineering evolution.

An Evolution is a first-class record of a coherent unit of engineering work:
intent, outcome, behavior, decisions, risks, verification, sessions, timeline,
relationships, and Git implementation references.

Git remains the source of truth for code. EVE complements Git by preserving the
engineering history around that code.

## Core Thesis

Git answers: how was this implemented?

EVE answers: how did the software evolve?

## Protocol Boundaries

EVE specifies:

- Evolution representation
- canonical JSON serialization
- field semantics
- relationship semantics
- lifecycle semantics
- Git implementation references

EVE does not specify:

- storage
- synchronization
- auth
- review policy
- verification policy
- deployment
- UI

## Canonical Representation

Every Evolution MUST have a canonical JSON representation.

Other formats such as YAML, TOML, SQLite, or protobuf MAY be used if they
round-trip losslessly to canonical JSON.

## Top-Level Shape

```json
{
  "eve": {
    "version": 1
  },
  "metadata": {},
  "intent": "",
  "outcome": "",
  "behavior": {},
  "decisions": [],
  "risks": [],
  "verification": [],
  "sessions": [],
  "timeline": [],
  "relationships": {},
  "implementation": {},
  "extensions": {}
}
```

## Metadata

```json
{
  "id": "ev_182",
  "title": "Enterprise SSO",
  "type": "feature",
  "status": "active",
  "created_by": "claude-code",
  "created_at": "2026-07-02T12:00:00Z",
  "updated_at": "2026-07-02T13:00:00Z"
}
```

Status enum:

- `draft`
- `active`
- `completed`
- `archived`

Type SHOULD be one of:

- `feature`
- `bugfix`
- `refactor`
- `migration`
- `security`
- `performance`
- `maintenance`
- `documentation`
- `infrastructure`

Producers MAY define additional types.

## Behavior

```json
{
  "added": [
    {
      "description": "Organizations can authenticate using Okta",
      "evidence": {
        "commits": ["81ab92"],
        "files": ["auth/providers/okta.ts"],
        "tests": ["auth.okta.test.ts"]
      }
    }
  ],
  "changed": [],
  "removed": [],
  "fixed": []
}
```

Behavior is declarative. EVE records claims; it does not verify them.

`outcome` SHOULD be consistent with behavior. Consumers MAY surface substantial
divergence.

## Verification

```json
[
  {
    "type": "unit-tests",
    "status": "passed",
    "reference": "npm test"
  }
]
```

Verification statuses:

- `passed`
- `failed`
- `skipped`
- `pending`
- `generated`
- `approved`

## Sessions

```json
[
  {
    "provider": "claude-code",
    "id": "session_912",
    "uri": "claude://session/912"
  }
]
```

Session IDs are opaque. Resolution is provider-specific.

## Timeline

Timeline entries are append-only.

```json
[
  {
    "timestamp": "2026-07-02T12:10:00Z",
    "actor": {
      "type": "agent",
      "provider": "claude-code",
      "id": "session_912"
    },
    "event": "implementation_started",
    "description": "Started implementing SAML provider support."
  }
]
```

## Relationships

```json
{
  "extends": ["ev_140"],
  "depends_on": ["ev_102"],
  "corrects": ["ev_121"],
  "supersedes": ["ev_088"],
  "reverts": ["ev_077"],
  "related": ["ev_099"]
}
```

Semantics:

- `extends`: adds work while preserving the original Evolution.
- `depends_on`: requires another Evolution before completion.
- `corrects`: fixes errors without removing the original outcome.
- `supersedes`: replaces the preferred engineering approach.
- `reverts`: removes the outcome introduced by another Evolution.
- `related`: association without semantic meaning.

Conflicting relationships are permitted. Consumers MAY surface conflicts rather
than resolving them silently.

## Implementation

Implementation belongs to Git.

```json
{
  "repositories": {
    "web": {
      "status": "merged"
    },
    "api": {
      "status": "merged"
    }
  },
  "commits": ["81ab2f", "e98d11"],
  "pull_requests": ["412"],
  "files_changed": 214,
  "insertions": 8421,
  "deletions": 3122
}
```

Implementation fields SHOULD be derived from Git.

An Evolution MUST NOT transition to completed while any referenced repository
remains non-terminal.

Git commits MAY be referenced by multiple Evolutions. Evidence references SHOULD
be scoped to the behavior claim they support. Consumers MAY surface repeated
commits cited for unrelated claims across Evolutions.

## Declarative vs Deterministic

Declarative fields:

- intent
- outcome
- behavior
- decisions
- risks
- relationships
- sessions
- timeline
- verification descriptions
- type

Deterministic fields:

- commits
- repositories
- pull requests
- files changed
- insertions
- deletions

## Extensibility

Producers MAY include extension fields.

Consumers MUST preserve unknown fields.

Consumers MUST ignore unknown fields they do not understand.

Extensions SHOULD be namespaced:

```json
{
  "extensions": {
    "acme": {
      "rollout": "25%",
      "feature_flag": "enterprise-sso"
    }
  }
}
```

## Historical Semantics

Completed Evolutions are immutable.

Future work creates new Evolutions that may extend, correct, supersede, or
revert earlier Evolutions.

Consumers rendering current state SHOULD traverse relationship graphs and prefer
newer Evolutions that correct, supersede, or revert earlier ones.

## Closing

Git preserves implementation history.

EVE preserves engineering history.

Together, they describe both how software was implemented and how it evolved.
